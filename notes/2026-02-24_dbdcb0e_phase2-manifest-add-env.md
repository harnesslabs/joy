# Phase 2 Manifest Parsing, `joy add`, and Local Environment

## Why

Phase 2 is the bridge from a pure project scaffolder to an actual package manager workflow. Implementing a typed `joy.toml` schema and `joy add` mutation logic establishes the core state model that all later resolution/fetch/build phases will consume. The local `.joy` directory layout is also foundational for future include/lib installation and build outputs.

## Blockers & Solutions

- Blocker: `joy add` needed deterministic manifest serialization for tests, but TOML emitters can change formatting compared to the template.
- Solution: Used typed serde models with `BTreeMap` and `toml::to_string_pretty`, and updated tests to assert the actual serialized shape generated by the tool.

- Blocker: Phase 1 tests expected `add` to be a stub, which became invalid once Phase 2 implemented real behavior.
- Solution: Split `add` coverage into dedicated integration tests and reduced the stub assertions to `build`/`run` only.

## Fallback Plan

If the typed `Manifest` model becomes too rigid for comment-preserving edits later, keep it for validation/roundtrip and add a separate lower-level TOML patching path for user-facing formatting preservation. The current command integration is isolated in `src/commands/add.rs`, so the mutation backend can be swapped without changing CLI behavior.
